## Диагноз (уже нашёл, экономь время на этом)
`crates/app/src/notifications/mod.rs:31-35`:
```rust
const POPUP_WIDTH: f32 = 360.;
/// card and let the compositor clip overflow.
const POPUP_HEIGHT: f32 = 96.;
```
Окно layer-shell создаётся с ФИКСИРОВАННОЙ высотой 96px и никогда не
меняется. `notifications/view.rs` рендерит карточки без ограничения
высоты — заголовок + summary (bold) + body (без line-clamp, переносится
на сколько нужно строк) + опциональные кнопки действий + при нескольких
одновременных уведомлениях они складываются в `flex_col` стопкой. Любое
уведомление с длинным `body` или больше одного уведомления разом —
контент выше 96px, а окно не резинится → обрезка, ровно то, что видит
пользователь. Комментарий «let the compositor clip overflow» — это был
осознанный, но неверный выбор в исходном дизайне.

2318, реализация в gpui_linux/src/linux/wayland/window.rs:1468) — ищи
живые примеры вызова в дереве (если их пока нет — будешь первым, ничего
страшного). Точка, откуда есть `&mut Window` — `sync_window` в
`crates/app/src/notifications/mod.rs` (там уже есть `handle.update(cx,
|_, window, _| ...)` паттерн, смотри рядом с `remove_window()`).

Варианта два, выбери и обоснуй в отчёте:
1. **Честный resize** — посчитай/оцени высоту контента (кол-во строк
   каждой карточки × line-height + паддинги + кнопки, если есть) и
   вызови `window.resize()` при каждом изменении `NotificationPopupState`

сервис `crates/services/src/tray/`, виджет `crates/app/src/bar/widgets/
tray.rs`, меню `crates/app/src/tray_menu/` (твоё №10).

**Симптом:** правый кластер бара забит ~13 одинаковыми безликими
глифами — визуально «частокол микрофонов», сжирающий половину бара.
Живьём подтверждено grim'ом.

**Причина установлена, это НЕ наш баг по происхождению:** Vivaldi
(Chromium) регистрирует новый `StatusNotifierItem` и не снимает
регистрацию, пока браузер жив. Проверено на шине:

```
busctl --user call org.kde.StatusNotifierWatcher /StatusNotifierWatcher \
  org.freedesktop.DBus.Properties Get ss \
  org.kde.StatusNotifierWatcher RegisteredStatusNotifierItems
  → 13 штук chromium
```

В логе шелла они капают по одной каждые пару минут:
```
tray: item added: :1.75/org/chromium/StatusNotifierItem/15
      (title=Some(""), icon=None, menu=Some("/org/chromium/DbusMenu/15"))
```

Все с `icon=None` И пустым `title` — опознать их невозможно, клик
бессмыслен. Наш `remove_item` (`tray/mod.rs:280`) срабатывает только
когда с шины пропадает ВСЁ имя (`:1.75`, `tray/mod.rs:487`) — а оно
живо, пока жив браузер. Формально мы отображаем ровно то, что
зарегистрировано; фактически бар превращается в кладбище глифов.

## Задача — защита на нашей стороне (решение пользователя: чинить у нас)

Три меры, все три:

1. **Фильтр бесполезных.** Item без иконки И без непустого `title` не
   рендерится вообще. Обоснование: опознать нечем, клик бессмыслен.
   Фильтровать в ВИДЖЕТЕ (`bar/widgets/tray.rs`) — сервис пусть держит
   правду шины целиком (она нужна для меню/отладки).
2. **Дедуп по владельцу шины.** Несколько item'ов от одного bus-имени
   (`:1.75`) → один значок. Показывай первый пригодный (с иконкой/
   title); остальные схлопываются. Клик/меню — по показанному.
3. **Кап.** Не больше `MAX_TRAY_ITEMS` (8) значков; сверх того —
   компактное `+N` тем же визуальным языком, что бейдж updates/bell
   (токены, `text.muted`).

Порядок применения: фильтр → дедуп → кап.
