# Задание №8 — настоящая причина ghost-окна лаунчера (re-entrant close), не Source

**Контекст (холодная сессия):** ChronOS, master. ПЕРВОЕ чтение —
HANDOFF.md, раздел «СИСТЕМНЫЙ БАГ remove_window» (переписан заново
после этого расследования — читай АКТУАЛЬНУЮ версию, не старую). Твоё
№7 (launcher click-fix) принято. Grok чинил `Source/` (форк gpui) —
задание №6, коммит `3800d3a` — это РЕАЛЬНЫЙ и полезный фикс (sync
flush+unregister в `Drop for WaylandWindow`), но он НЕ закрывает баг
целиком: я живьём поймал, что для лаунчера окно остаётся ghost'ом
ДАЖЕ С этим фиксом. Причина — в твоём модуле, не в Source.

## Что я нашёл (проверено живьём, не гипотеза)

Открыл лаунчер через IPC toggle. Окно само потеряло активность
(`is_window_active=false`) естественным образом ~550мс спустя (не мой
клик — фокус ушёл сам, вероятно compositor). Лог:
```
launcher activation observer fired active=false was_active=true interacted=false
launcher::close called
launcher handle taken, removing window
```
После этого `hyprctl clients -j` **не переставал показывать окно
`mapped:true`** 6+ секунд подряд (поллинг 100-200мс). Диагностический
лог Грока (`Drop WaylandWindow`, `drop_window done`, `flush after
destroy ok` — из `RUST_LOG=gpui_linux=debug`) НИ РАЗУ не появился.
Т.е. `window.remove_window()` физически НЕ ВЫЗЫВАЕТСЯ в этом пути —
Grok'ов фикс в Source просто не успевает сработать, потому что до
него дело не доходит.

## Корень (прочёл код `app.rs::update_window_id`, Source, не трогать
её — это ChronOS-баг, не Source-баг)

`App::update_window_id(id, f)` делает `cx.windows.get_mut(id)?.take()?`
— забирает `Box<Window>` из слота НА ВРЕМЯ выполнения `f`. Пока слот
пуст (мы «внутри» обновления этого окна), ЛЮБОЙ повторный
`update_window_id` на ТОТ ЖЕ id вернёт `Err("window not found")` —
слот пуст, `.take()` вернёт `None`.

`cx.observe_window_activation` (твой файл, `launcher/mod.rs:102`)
вызывается ИЗНУТРИ такого обновления — смотри
`Source/gpui/src/window.rs:1589-1608`
(`on_active_status_change` → `handle.update(&mut cx, |_, window, cx| {
... activation_observers.retain(...) ...})`). То есть твой callback
уже выполняется "внутри" забронированного слота окна лаунчера.

Твой `close_this()` (`launcher/mod.rs:155`) получает `window: &mut
Window` НАПРЯМУЮ (он уже у тебя в руках!), но при `tracked=true` не
использует его — зовёт `close(cx)` (строка 164), а тот делает ЕЩЁ ОДИН
`handle.update(cx, |_, window, _| window.remove_window())` (строка
143) — это РЕЕНТЕРАБЕЛЬНЫЙ вызов на тот же window id, слот занят,
`Err("window not found")`, и `let _ =` (строка 143) ЭТУ ОШИБКУ МОЛЧА
ГЛОТАЕТ. `window.remove_window()` **никогда не вызывается**. Global
`LauncherState.handle` при этом всё равно очищается (`.take()` в
`close()` срабатывает раньше, до `handle.update`), поэтому `is_open`
на следующий toggle снова видит `false` и создаёт НОВОЕ окно — а
старое остаётся висеть в компоузиторе навечно (ghost).

Тот же антипаттерн — подозреваю — в `tray_menu::click_item`
(`crates/app/src/tray_menu/view.rs:178`, Autohand, ещё в поле): его
`on_click` получает `_window` и ИГНОРИРУЕТ его, зовёт `click_item(cx,
id)` → `close(cx)` → тот же реентерабельный `handle.update`. Это НЕ
твоя зона, не трогай — Autohand разберётся сам, я его предупредил.

## Задача

Почини `close_this()` (и, если понадобится, `close()`) так, чтобы
закрытие ИЗНУТРИ уже идущего обновления окна (наблюдатель активации,
любой колбэк с `window: &mut Window` в сигнатуре) вызывало
`window.remove_window()` **напрямую на уже имеющейся ссылке**, а не
через новый `handle.update(cx, ...)`. Набросок направления (не готовый
код, реши сам с учётом реального API):
```rust
pub(crate) fn close_this(window: &mut Window, cx: &mut App) {
    let this = window.window_handle();
    let tracked = cx.global::<LauncherState>().handle.as_ref()
        .map(|h| **h == this).unwrap_or(false);
    if tracked {
        cx.global_mut::<LauncherState>().handle = None;
    }
    window.remove_window(); // напрямую, без повторного update_window_id
}
```
`close(cx: &mut App)` (вызывается СНАРУЖИ window-контекста — IPC,
autoclose-таймер, если есть) может остаться как есть (не
реентерабелен в этих путях — там `handle.update` первый и единственный
вызов на этот id).

Проверь ВСЕ места, откуда реально зовётся `close()`/`close_this()` — не
только те два, что я нашёл — грепни `close(` и `close_this(` по
launcher/**, для каждого места пойми: вызывается ли это ИЗНУТРИ
колбэка, у которого уже есть `&mut Window` того же окна (реентерабельно,
чини) или СНАРУЖИ (не трогай).

`follow_mouse = 1` (Lua-Hyprland). Это штатный Hyprland-режим
"клавиатурный фокус следует за курсором": стоит увести мышь с окна на

окну и `Enter` тому, что под курсором — БЕЗ клика. Это долетает до
`Source/gpui_linux/src/linux/wayland/client.rs:1594-1610`

`active: true→false` — ровно тот сигнал, который твой №8-фикс (верно)

`Source`-цепочке НЕТ отдельного mouse-move/mouse-leave хендлера —
это НАСТОЯЩАЯ потеря клавиатурного фокуса, просто вызванная не
кликом, а политикой compositor'а `follow_mouse=1`.

Это НЕ регресс №8 — №8 чинил реентерабельность самого вызова
`remove_window()`, и чинил верно. Это отдельный, ранее не пойманный

задокументировать, почему нельзя).
