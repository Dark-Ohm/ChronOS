## Текстовый ввод — `ccf-gpui-widgets` (разведано 2026-07-20)

Самая продуктово значимая находка сегодняшнего дня — закрывает
named gap, а не абстрактную возможность. `DECISIONS.log` 2026-07-16
отказался от текстового ввода Kael как overkill («2987+243 LOC +
отсутствующий undo_manager… launcher-ввод добить руками») и оставил
заглушку. Boковая agent-панель с чатом (см. выше) без многострочного
текстового поля с курсором/историей/фокус-навигацией не строится.

`ccf-gpui-widgets` (18000 строк, v0.1.0) даёт ровно это:
`text_input.rs`/`cursor_blink.rs`/`editing_core.rs`/`focus_navigation.rs`
+ россыпь готовых виджетов (`file_picker`, `dropdown`, `slider`,
`toggle_switch`, `confirmation_dialog` и др. — не всё нужно, но список
широкий). `gpui = "0.2.2"` голый (версия совпадает с нашим форком),
`smol` внутри — паттерн executor'а не проверен (тот же класс вопроса,
что у `gpui-animation`).

**Трение — их собственная плоская `Theme` (`u32`-поля:
`bg_primary`/`text_primary`/…), не `gpui-component`.** Дешевле, чем
казалось: структура семантически близка нашей, виджеты принимают
`custom_theme: Option<Theme>` явно — маппинг `chronos_ui::Theme` → их
`Theme` это одна функция на десяток полей, не архитектурная работа.

**Разведано компиляцией 2026-07-21 (Hermes) — вердикт: ВЕНДОР ОБЯЗАТЕЛЕН,
ОТЛОЖЕН.** НЕ компилируется против форка: 58 ошибок, 5 классов форк-дельт
(`FocusHandle::focus(+cx)` ×43, `update_entity→R` ×3, `Corner` удалён,
`Bounds::from_corner_and_size` удалён, `ScrollHandle::max_offset` сменил
тип). Animation-паттерн, не rsx — `[patch]`-зависимостью не подключить.
Ввод хирургически извлекаем (`text_input+editing_core+cursor_blink+
focus_navigation`, 4 файла), но дельты чинить ровно в них. Тема-маппинг
нетривиален: их 52 плоских `u32`-поля vs наши `Hsla`-группы → адаптер
`Hsla→u32` ~40 полей + коллизия `gpui::Global`. **Отложено:** named gap
реален, но не на критическом пути правой панели; вернёмся отдельной
задачей к agent-панели. Отчёт: `report-log/hermes-report-widgets-recon.md`.

## Кандидаты на переходы/анимацию — `gpui-animation` (разведано 2026-07-20)

Не привязано к панелям, применимо уже сегодня. `gpui = "0.2.2"` без
`gpui-component`, спит на нашем же GPUI executor (`cx.spawn(...)
.detach()` — сама библиотека соблюдает правило, которое мы весь день
ловили как баг у своих виджетов), `smol::Timer` — просто future внутри
таски, третьего рантайма не заводит. Даёт декларативный слой ПОВЕРХ
уже вшитых в форк `EasingCurve` (`gpui/src/easing.rs`, находка DeepSeek
№2): `.transition_on_hover(duration, curve, |state, style| ...)`,
умная интерполяция HSLA по кратчайшему пути, откат при прерывании одной
анимации другой. Прямой кандидат применения — hover шести виджетов
правого кластера бара (Mimo №11, сейчас мгновенный щелчок цвета через
голый `.hover(|s| s.bg(...))`) стал бы плавным затуханием без ручной
анимационной логики. Пометка: README называет себя «early development,
API subject to change» — готовиться к breaking changes при апдейте.

**Разведано компиляцией 2026-07-20 (Grok №18) — статус НИЖЕ, чем у
gpui-form/gpui-rsx.** НЕ компилируется out-of-the-box: 3 API-дельты
форка против crates.io `0.2.2` (`AsyncApp::update` non-fallible,
`BoxShadow.inset` — лишнее поле, `Style.text` `#[refineable]`).
После 3 хирургических правок их исходников (~15 строк) — компилируется
и живой hover-переход подтверждён grim (тёмный → промежуточный синий →
целевой голубой, не snap). **Правило для будущих разведок:** совпадение
семвера (`0.2.2` у обеих сторон) НЕ значит совместимость ABI, когда одна
сторона — git/path-форк, а другая — crates.io. Взять эту библиотеку =
взять на сопровождение форк её кода (patch+vendored tree), не просто
добавить строку в Cargo.toml.

**РЕШЕНИЕ: БЕРЁМ, ВЕНДОРНУТО 2026-07-21.** Крейт `Source/gpui-animation/`
(Grok, принят) — копия `@ad77bea` + 3 форк-дельты + `PATCHES.md` под
будущий апдейт, атрибуция в `Source/NOTICE`, проведён в `Source`
workspace.members, `cargo check` против форка зелёный (26с, 0 ошибок).
Следующий шаг (не начат): зависимость `crates/app` + пилот на hover
правого кластера бара (Mimo №11 — сейчас голый `.hover`).

## JSX-разметка — `gpui-rsx` (разведано и ПОДТВЕРЖДЕНО компиляцией, 2026-07-20)

`gpui-rsx` разведано компиляцией 2026-07-20 (Cline №2) — компилируется
против нашего форка без правок их кода, auto-`.id()` подтверждён с двух
сторон (их codegen + наша трейт-граница `StatefulInteractiveElement`).
Дисциплина использования зафиксирована: ТОЛЬКО individual attributes
(`bg={theme.bg.primary}`), `class="bg-*"` запрещён (хардкодит палитру
мимо `Theme::global`). Пилот на `system_popup/view.rs` (29 `div()`,
38 `.child(`) — не роздан.

**РЕШЕНИЕ: БЕРЁМ, во владение (вендор, не dep).** rsx — чистый
proc-macro (deps только syn/quote/proc-macro2, gpui-зависимости нет →
типизируется в потребителе), поэтому и компилился без правок. Вендор в
`Source/gpui-rsx/` роздан Cline — **исполнение завтра** (2026-07-21
поздно rsx-отчёта ещё не было). Обоснование вендора vs `gpui-rsx="0.6.0"`
одной строкой: владение (автор со слабым версионным контролем, китайская
диагностика — для шелла свой пиннутый форк надёжнее плавающего dep).

## Известные системные ограничения (не менять походя)

- ~~`overflow_y_scroll()` не резолвится в нашем форке gpui~~ **НЕВЕРНО,
  опровергнуто 2026-07-20** (нужен `.id()` — см. DECISIONS). Ранее считалось —
  только
  `overflow_hidden()` (клип, не скролл). См. ARCHITECTURE.md §4.1.
- `follow_mouse=1` — популярные попапы никогда не закрываются по потере
  фокуса, только явным действием. См. ARCHITECTURE.md §4.1.
- ydotool на этой машине нестабилен (dual-monitor calibration drift) —
  живые клик-смоки синтетическим вводом ненадёжны, финальную приёмку
  кликом делает пользователь лично.
