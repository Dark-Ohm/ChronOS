# Updates popup — anchored redesign (pilot for popup-system polish)

_2026-07-24. Первый из пяти попапов (`updates_popup`, `volume_popup`,
`system_popup`, `notifications/history_popup`, `tray_menu`) переводится
на новую дисциплину позиционирования/размера/визуала. Остальные четыре —
отдельные будущие задачи по этому же образцу, НЕ в этой спеке._

## Зачем

Пользователь назвал четыре конкретные проблемы текущих попапов:

1. Все 5 жёстко анкорены `Anchor::TOP | Anchor::RIGHT` независимо от
   того, где в баре находится триггер-иконка — попап всегда открывается
   в одном и том же углу экрана.
2. Скролл был отложен, не отменён — сейчас список режется `max_h()` +
   `overflow_hidden()` (см. `updates_popup/mod.rs` — защитная мера после
   реального инцидента 2026-07-19: непромеренная высота строки унесла
   кнопку "Upgrade all" за нижний край окна). Момент настал — список
   должен быть реально доскроллен, не обрезан.
3. Визуально — "MVP", нет полировки уровня остального шелла.
4. Окно не подстраивается под контент — фиксированная высота-прикидка
   (`estimate_popup_height`), не измерение реального контента.

## Пилот: `updates_popup`

Выбран как эталон — самый маленький (496 строк), один триггер в баре,
уже даёт список переменной длины (хороший тест и на скролл, и на
adaptive-размер). Остальные 4 попапа переводятся на этот образец
отдельными задачами после живой приёмки этого пилота.

## 1. Позиционирование — `WindowKind::AnchoredPopup`

Переход со standalone `WindowKind::LayerShell` (текущий, жёстко
`TOP|RIGHT`) на нативный anchored-popup механизм форка
(`gpui/src/platform/popup.rs`, skill `anchored-popups`).

- `parent: AnyWindowHandle` — окно бара.
- `anchor_rect: Bounds<Pixels>` — реальные bounds иконки-триггера
  (`bar/widgets/updates.rs`) в координатах бара. Сейчас `updates_popup::
  open(cx: &mut App)` НЕ принимает bounds вообще — сигнатура меняется на
  `open(cx: &mut App, anchor_rect: Bounds<Pixels>, parent: AnyWindowHandle)`
  (или эквивалент, финальную форму зафиксировать в плане).
- `anchor: PopupAnchor::BottomRight`, `gravity: PopupGravity::BottomRight`
  — попап падает вниз-вправо от иконки (бар у верхнего края экрана).
- `constraint_adjustment: FLIP_Y | SLIDE_X` — не должно уезжать за
  верхний край (там ему уезжать некуда, бар и так сверху) и не должно
  вылезать за правый край экрана для иконок у самого края.
- `grab: true` — это menu-like попап (дропдаун списка), должен брать
  клавиатурный фокус и закрываться по клику мимо.
- **Триггер переезжает с `on_click` на mouse-down** (`bar/widgets/
  updates.rs:81`) — grab-попапы должны запрашиваться, пока исходный ввод
  ещё активен (`anchored-popups` skill, "Common Mistakes #1") — на
  `on_click` compositor откажет в захвате.
- **`grab` покрывает dismiss только для кликов в ДРУГИЕ приложения.**
  Клик мимо попапа, но внутри ChronOS (например, по бару) — по-прежнему
  наша забота, дисциплина dismiss остаётся той же, что и сегодня (Esc,
  крестик, повторный клик по триггеру, "Upgrade all") + добавляется явный
  outside-click-внутри-своего-приложения хендлер.
- **Fallback:** `PopupNotSupportedError` (не все платформы имеют нативный
  popup) → откат на текущий standalone `LayerShell` `TOP|RIGHT` путь, не
  hard-fail. Практически на Hyprland/Wayland должно работать нативно —
  живой смок подтверждает, fallback-ветка тестируется хотя бы компиляцией.

## 2. Размер и скролл

Header и footer остаются вне скролла — тот же защитный принцип, что и
сегодня (инцидент 2026-07-19: непромеренная высота = невидимая кнопка),
просто список внутри становится **реальным** `overflow_y_scroll()` с
`.id()` (подтверждено рабочим в форке с 2026-07-20 — старое "не
резолвится" опровергнуто) вместо `overflow_hidden()`-обрезки.

- `HEADER_BUDGET_H`/`FOOTER_BUDGET_H` — остаются фиксированным бюджетом
  (не измеряются, тот же принцип что и сейчас).
- `LIST_MAX_H` — потолок списка, **300→340px** по мокапу
  (`design/Updates Popup.dc.html`).
- `MAX_POPUP_H` = `HEADER_BUDGET_H + DIVIDER_H + LIST_MAX_H +
  FOOTER_BUDGET_H` — та же формула, что и сейчас, просто список внутри
  скроллится, а не пропадает: если пакетов больше, чем влезает в 340px,
  их можно доскроллить — то, что раньше буквально терялось за обрезкой.
- Список — `div().id("updates-popup-list").overflow_y_scroll()
  .track_scroll(&self.scroll)` (builder, не rsx — динамический список с
  `cx.listener` на каждой строке, см. §3).

## 3. Визуал — 1:1 по мокапу `design/Updates Popup.dc.html`

Dark (эталон) + Light (принятый "Light C") варианты, оба заданы в
мокапе буквально (хексы, тени, watermark-сигил, glow-линия — канон,
не выдумывать).

**rsx vs div map** (обязателен в отчёте по завершении, дисциплина
`gpui-rsx`):

| Элемент | Подход | Причина |
|---|---|---|
| Header (заголовок + крестик закрытия) | `rsx!` | статичная структура, простой `onClick` |
| Watermark-сигил + glow-линия (light-вариант) | `rsx!` | статичная декорация, условный рендер по теме |
| Список пакетов | builder `div()` | динамический, per-row hover, `overflow_y_scroll` + `track_scroll`, `key` на каждой строке (AUR-пилюля, версии) |
| Footer "Upgrade all" | `rsx!` | статичная структура, простой `onClick` |

Цвета — `rgb(0xRR_GG_BB)` литералы из мокапа для этого прохода (пиксель-
паритет), НЕ `Theme::global(cx)` токены — тот же режим, что sidebar v2
(мокап-парность важнее токенизации в первом проходе; токенизация — 
отдельный будущий пass, если решим).

## Верификация

Живой смок обязателен (release + `RUST_LOG=info`, live grim — попапы это
window/UX-код, "компилируется и тесты зелёные" не считается):

1. Клик по иконке updates в баре — попап открывается ВОЗЛЕ иконки
   (не в фиксированном углу), в правильную сторону (вниз-вправо,
   не обрезан верхним краем).
2. Список короче потолка (1-2 пакета) — окно НЕ раздувается до
   максимума, содержимое не обрезано.
3. Список длиннее потолка (7+ пакетов, как в мокапе) — реально
   доскроллить до последней строки, "Upgrade all" остаётся видна и
   кликабельна.
4. Клик мимо попапа (по бару, по десктопу) — закрывается.
5. Esc — закрывается.
6. Dark и Light темы — оба варианта визуально сверены с мокапом (grim
   скрины обоих).
7. Иконка-триггер у самого правого края экрана (если применимо к layout
   бара) — попап не вылезает за экран (`SLIDE_X` отрабатывает).

## Self-review

**Placeholder scan:** нет TBD/TODO — открытый пункт (точная сигнатура
`open()`) явно помечен "финализировать в плане", не оставлен неявным.
**Внутренняя согласованность:** формула `MAX_POPUP_H` не меняется,
меняется только механизм внутри списка (clip→scroll) и позиционирование
окна (fixed corner→anchored) — не противоречит текущему защитному
принципу (footer вне скролла), усиливает его.
**Скоуп:** сфокусировано на одном попапе, явный "что не делать" список
отсекает остальные 4 попапа и смежные фронты (bar-layout, Plugin API v2).
**Неоднозначность:** grab-дисциплина (dismiss только для чужих приложений,
свои клики — наша забота) явно расписана, не оставлена на усмотрение
исполнителя.
